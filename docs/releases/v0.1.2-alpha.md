# SilkMC v0.1.2-alpha

This release focuses on massively improving plugin compatibility with the Paper, Bukkit, and Spigot ecosystems while preserving Folia's regional multithreading performance.

## Compatibility improvements

- **Scheduler bridging**: Legacy Bukkit sync scheduler calls (`runTask`, `runTaskLater`, `runTaskTimer`, `callSyncMethod`) are now safely bridged to the Global Region Scheduler instead of throwing `UnsupportedOperationException`.
- **Teleport compatibility**: Legacy synchronous `Entity.teleport()` and `Player.teleport()` calls are bridged to `teleportAsync` when safe. Cross-region sync teleports log a warning and return optimistically to avoid breaking plugins.
- **Plugin loading policy**: Unmarked plugins (those without `silk-supported: true` or `folia-supported: true`) can now load in warning mode by default. A configurable compatibility policy controls whether to warn, require explicit metadata, or auto-disable on failure.
- **silk-supported metadata**: Plugins can declare `silk-supported: true` in `plugin.yml`. Both `silk-supported` and legacy `folia-supported` are honored.
- **Thread compatibility**: `isPrimaryThread()` continues to return true for any server tick thread, letting plugins proceed while region ownership checks preserve correctness.

## Stability and safety

- Unsafe operations that cannot be bridged still fail gracefully with clear operator-facing log messages.
- Compatibility failure detection classifies known unsafe boundaries (world access from wrong thread, teleport assumptions, legacy scheduler paths) and can auto-disable offending plugins.
- Detailed compatibility stack traces are available via `silkmc-compatibility.yml`.

## Branding

- Startup banners, build metadata, version fetcher, and metrics now report `SilkMC` instead of upstream names.
- Upstream origins are mentioned only where legally required.

## Known limitations

- No fake global main thread is introduced. Plugins that mutate cross-region state unsafely can still fail.
- Some traditional Bukkit scheduler tasks still lack enough location or entity context for SilkMC to infer a safe owning region; these are routed to the global region scheduler, which may serialize more than ideal.
- Some event ordering assumptions from traditional single-threaded servers remain impossible to preserve fully.
- Benchmark numbers are still pending broader validation workloads.
