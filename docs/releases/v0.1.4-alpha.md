# SilkMC v0.1.4-alpha

This release introduces the **SilkPluginCompatibilityManager** — a pre-load JAR inspection and classification system that decides what to do with each plugin before it is enabled. It builds on v0.1.3's scheduler bridge fix.

## What's new

### Plugin Compatibility Manager

Every plugin JAR is now inspected before it is enabled. The manager:

1. **Inspects** the JAR — reads `plugin.yml` / Paper plugin metadata and scans all class bytes for known API usage patterns and NMS references.
2. **Classifies** the plugin into one of four buckets:
   - **SAFE** — declares `silk-supported: true` or `folia-supported: true`. Loads normally.
   - **COMPATIBLE** — uses Bukkit / Spigot / Paper APIs but does not show unsafe patterns. Loads under the SilkMC compatibility shim layer (scheduler bridge, teleport bridge, lifecycle failure classifier).
   - **UNSAFE** — combines async scheduling with direct NMS-level world access. Blocked at load time.
   - **UNKNOWN** — no APIs detected, JAR could not be read, or the profile is otherwise inconclusive. Blocked under strict-mode, allowed under `allow-unknown-plugins: true`.
3. **Decides** whether to load normally, load with shims, or block — and prints a structured rejection log for blocked plugins (plugin name, status, reasons, technical details, impact, recommendation).
4. **Activates shims** for COMPATIBLE plugins (scheduler bridge, teleport bridge, lifecycle classifier — see v0.1.2 release notes).

### Components

The manager consolidates the modular components described in the spec into a single orchestrator file (`gg.tame.silkmc.server.compat.SilkPluginCompatibilityManager`):

- `inspectAndEnforce(...)` - public entry points called from `PaperPluginProviderFactory.build()` and `SpigotPluginProviderFactory.build()`.
- `Findings` - record of what bytecode scanning detected.
- `Classification`, `NmsLevel`, `Family` - enums.
- `Report` - immutable result with reasons, technical details, scanned class count.
- Plus internal: scanner (`scanSafely`), bytecode analyzer (`analyzeClassBytes`), classifier (`classify`), decision engine (`enforce`), shim manager (`activateShim`), and structured rejection log formatter (`logRejection`).

### Configuration

`silkmc-compatibility.yml` adds a new section:

```yaml
compatibility:
  strict-mode: true
  allow-unknown-plugins: false
  log-analysis: true
```

- `strict-mode` blocks UNKNOWN plugins automatically.
- `allow-unknown-plugins` overrides strict-mode for UNKNOWN plugins (load with warning).
- `log-analysis` prints a one-line summary per inspected plugin.

The existing `plugins:` section is unchanged.

### Structured rejection log

When the manager blocks a plugin, it prints a single multi-line entry with the format:

```
[SilkMC Plugin Compatibility System]

Plugin: <name> v<version>

Status: BLOCKED (UNSAFE | UNKNOWN)

Reason(s):
- <issue 1>
- <issue 2>

Technical Details:
- <detail 1>
- NMS access level: <NONE|LOW|MEDIUM|HIGH>
- Plugin family: <PAPER|SPIGOT|BUKKIT|UNKNOWN>
- Scanned classes: <count>

Impact:
This plugin could cause world corruption, memory corruption, or thread safety violations under SilkMC.

Recommendation:
- Use a SilkMC-compatible build of the plugin
- Remove unsafe async world access
- Update plugin to Paper-compatible API usage
```

The server continues startup after a block; it does not terminate.

## Safety guarantees

The manager is conservative by design:

- It will not attempt runtime bytecode patching of plugins.
- It will not disable thread-safety checks.
- It defaults to BLOCK for plugins it cannot classify, when strict-mode is on.
- It preserves the region-thread safety model end to end.

## Backwards compatibility

- The legacy `silk-supported` / `folia-supported` metadata path is preserved. Marked plugins are classified SAFE and skip the heuristic rules.
- The v0.1.2 scheduler bridge, teleport bridge, and lifecycle failure classifier are still in place and continue to bridge legacy plugin behavior at runtime.
- `silkmc-compatibility.yml` from v0.1.3 keeps working; the new `compatibility:` section is added on first startup if missing.

## Known limitations

- Bytecode inspection uses byte-string scanning of the class constant pool rather than full bytecode analysis. This catches package-level and method-name patterns but does not perform data-flow analysis. The trade-off is zero new runtime dependencies and very fast scanning.
- The classifier intentionally errs on the side of COMPATIBLE for plugins that only use surface Bukkit/Paper APIs. Specific unsafe patterns (async + HIGH-NMS) are required to flag UNSAFE.
- Plugins whose JARs cannot be opened are classified UNKNOWN.

## Build & upgrade

Drop the new paperclip jar in place of the v0.1.3-alpha jar. Restart the server. The new `compatibility:` section will be appended to `silkmc-compatibility.yml` on first startup. No data migration required.
